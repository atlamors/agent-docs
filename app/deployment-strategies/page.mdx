# Agent Deployment Strategies

This section compares deployment models for Hermes-based agents that run through Slack, Telegram, and customer business systems.

The goal is narrow: choose where the runtime, customer data, RAG, connectors, model inference, logs, and update path should live for a given customer deployment.

This section does not define the vendor's product roadmap, pricing, customer success model, or support operating model. Where those assumptions affect deployment choice, they are marked explicitly in the [Technical Appendix](technical-appendix).

## Control Boundary Terms

- **Vendor:** the agent platform company operating the Hermes-based system.
- **Customer:** the client organization using the agent.
- **Vendor-managed cloud:** infrastructure owned or operated by the vendor.
- **Customer-specific vendor environment:** a vendor-managed environment dedicated to one customer.
- **Customer-controlled infrastructure boundary:** infrastructure, account, network, or hardware controlled by the customer.

## Agent Isolation Invariant

No deployment pattern should rely on a shared, unsandboxed agent runtime across customers. Runtime, credentials, tools, memory, retrieval, logs, and job execution must be sandboxed per customer.

Kubernetes namespaces, containers, VMs, VPSs, VPCs, and cloud accounts are implementation choices inside a deployment pattern. They do not create new patterns by themselves.

## Known Deployment Inputs

- Agents are deployed through Slack, Telegram, or similar channels.
- Agents need controlled access to customer systems such as Notion, knowledge bases, CRMs, project tools, and RAG-backed data.
- Customer deployments may require department-level permissions.
- Deployments need an update path across instances.
- Some customers may require local inference.
- Some customers may require full on-prem or bare-metal deployment.

## Option Summary

| Strategy | Runtime location | Customer data / RAG location | Model inference | Use when |
| --- | --- | --- | --- | --- |
| [Managed Cloud Runtime](managed-cloud-runtime) | Vendor-managed cloud, per-customer sandbox | Customer APIs or vendor-hosted customer-specific knowledge store | Cloud model | Standard cloud deployment with low customization. |
| [Isolated Cloud Tenant](isolated-cloud-tenant) | Customer-specific vendor environment | Dedicated customer-specific knowledge store, customer APIs, or both | Cloud model, with local add-on possible | Customer needs stronger isolation but not full on-prem. |
| [Hybrid Local Inference](hybrid-local-inference) | Vendor-managed cloud by default | Vendor-managed or customer-controlled, depending on boundary requirement | Customer-controlled inference endpoint | Customer wants local model execution. |
| [Full On-Prem Appliance](full-on-prem-appliance) | Customer-controlled infrastructure boundary | Customer-controlled infrastructure boundary | Customer-controlled infrastructure boundary | Runtime, data, and inference must remain customer-controlled. |

## Selection Logic

- Start with Managed Cloud Runtime when vendor-managed cloud is acceptable and the customer needs a standard sandboxed deployment.
- Use Isolated Cloud Tenant when the customer needs a stronger customer-specific vendor environment for runtime, secrets, logs, retrieval, or connectors.
- Use Hybrid Local Inference when the customer wants model execution to happen on customer-controlled hardware, but does not require the full agent stack to move local.
- Use Full On-Prem Appliance when runtime, retrieval, connectors, logs, and inference must all run inside the customer-controlled infrastructure boundary.

## Boundary Questions

These questions determine the deployment strategy.

- Can customer data be processed in vendor-managed cloud infrastructure?
- Can customer documents be indexed outside the customer-controlled infrastructure boundary?
- Does the customer require local inference, or full local processing?
- Are Slack, Telegram, Notion, CRM, KB, and project-tool integrations available through APIs?
- Are department permissions enforced in source systems, the agent layer, or both?
- What logs, traces, and audit events may leave the customer-controlled infrastructure boundary?
- Does the customer require isolated runtime, isolated knowledge storage, isolated secrets, or all three?
- How are runtime, connector, policy, and model-routing updates applied?
- What rollback path is required if an agent update or connector update fails?

## Unit Economics Inputs

Deployment strategy changes the cost to launch, operate, update, and support each customer instance.

This section does not assume pricing or margin. It identifies the cost inputs that should be measured before choosing a default deployment model.

| Cost driver | Why it affects deployment choice |
| --- | --- |
| Initial setup time | Determines whether a strategy can be repeated without custom engineering. |
| Runtime infrastructure | Customer-specific vendor environments and on-prem deployments add fixed per-customer overhead. |
| Model inference cost | Cloud models, local inference units, and customer-hosted models create different cost profiles. |
| Connector maintenance | API-backed deployments depend on connector stability, auth refresh, rate limits, and schema drift. |
| Knowledge maintenance | Customer-specific knowledge stores require ingestion, refresh, deletion handling, and permission checks. |
| Observability and support access | More restricted deployments can increase time-to-debug and rollback effort. |
| Update and rollback effort | A strategy is more expensive if every upgrade requires customer-specific coordination. |
| Hardware or appliance handling | Hybrid local inference and on-prem deployments add procurement, installation, monitoring, and replacement work. |

The appendix includes an assumption-based checklist for tracking these costs by deployment mode.
