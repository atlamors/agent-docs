# Dedicated Cloud Tenant

Dedicated Cloud Tenant is the recommended default for bespoke paid deployments.

Each customer gets an isolated VPS or tenant cell containing the customer-specific runtime, secrets, RAG, queues, logs, and connector workers.

```txt
Customer channels
  -> dedicated customer VPS / tenant cell
     - Hermes runtime
     - customer RAG
     - channel adapters
     - connector workers
     - task queue
     - logs and audit
     - customer secrets
  -> model gateway
```

## Use When

- The customer is paying for a bespoke install.
- The deployment has meaningful operational or security expectations.
- The team needs clear per-customer cost accounting.
- Support and debugging need to be isolated.
- The customer is not ready for on-prem or local inference.

## What Runs Where

| Component | Location |
| --- | --- |
| Hermes runtime | Customer-specific VPS or tenant cell |
| Slack / Telegram adapters | Customer-specific environment |
| RAG index | Customer-specific environment |
| Secrets | Customer-specific vault or secret store |
| Model gateway | Shared or tenant-scoped |
| Observability | Centralized, tenant-tagged |

## Strengths

- Strong isolation without full on-prem burden.
- Cleaner debugging than multi-tenant cloud.
- Good enterprise-ish story for SMB and mid-market buyers.
- Better cost visibility per customer.
- Still supports fleet-level updates.

## Risks

- More infrastructure overhead than Cloud Lite.
- Requires deployment automation early.
- If each tenant becomes hand-crafted, the model collapses into services work.
- Tenant sprawl can become painful without a release registry and health dashboard.

## Update Model

Dedicated tenants should be enrolled into release channels:

- internal
- canary
- beta
- stable
- pinned
- emergency rollback

Each tenant should report:

- Hermes runtime version
- agent-template versions
- connector versions
- model policy
- eval-pack status
- last update result
- rollback target

## Productization Path

This is the bridge from bespoke installs to SaaS.

The manual work performed during a Dedicated Cloud Tenant install should become:

- internal deployment automation
- admin configuration pages
- guided integration setup
- billing-tier controls
- rollout and health dashboards
