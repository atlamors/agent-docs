# Agent Docs

This docs set focuses on deployment strategies for Hermes-based agents deployed through Slack, Telegram, and adjacent business systems.

## Business Strategy

The company should use high-touch deployments to build a repeatable product, not a services-heavy implementation business.

Customers may need different deployment boundaries, integrations, permissions, and workflows. Those differences should be handled through a standardized deployment engine and reusable configuration, not one-off engineering.

The core product should remain consistent across every customer:

- one release process
- one update and rollback path
- one observability model
- one deployment harness
- one configuration system

After deployment, the customer-specific work should be captured as reusable configuration: workflows, tools, permissions, knowledge sources, model routing, escalation paths, and acceptance tests.

This creates a compounding business model. Each paid deployment should produce reusable assets that make the next deployment faster, cheaper, and more reliable.

Over time, those reusable configurations become the SaaS product: internal automation first, then preset deployment packages, then customer-facing GUI controls.

The strategic goal is to turn bespoke installation work into product leverage.

## Runtime Scaling Assumption

Each deployment pattern should support scaling Hermes behind a stable customer-facing entrypoint.

The customer should not need a new Slack or Telegram bot because one runtime is saturated. The deployment should expose one stable ingress, then add Hermes runtime replicas, orchestration capacity, and worker capacity behind that boundary.

This matters because Hermes capacity is not only message volume. A runtime may also be limited by how many workflows, spawned agents, tool calls, retries, and state transitions it can coordinate reliably at once.

The implementation may differ by deployment pattern:

- Managed Cloud Runtime scales replicas inside the vendor-managed cloud platform.
- Isolated Cloud Tenant scales replicas inside the customer-specific vendor environment.
- Hybrid Local Inference scales cloud orchestration separately from the customer-controlled inference endpoint.
- Full On-Prem Appliance scales within the customer-controlled infrastructure boundary.

The open measurement is the same across patterns:

> How much concurrent orchestration can one Hermes runtime safely handle before another replica is needed?
